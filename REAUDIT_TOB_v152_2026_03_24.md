### FIX-02 | ToB-001: Timestamp as_nanos u128->u64 Clamping | VERIFIED CORRECT

**File:** `crates/hashgraph/src/dag.rs` lines 208-217
**Original finding:** `as_nanos()` returns u128, bare cast to u64 truncates after ~584 years (2554 CE). Also, clock failure produces 0 which rejects all future events.

**Verification:**
```rust
let now_ns = std::time::SystemTime::now()
    .duration_since(std::time::UNIX_EPOCH)
    .unwrap_or_default()
    .as_nanos()
    .min(u64::MAX as u128) as u64;
let now_ns = if now_ns == 0 { event.timestamp_ns } else { now_ns };
```
- `.min(u64::MAX as u128)` clamps before cast — no truncation possible.
- `unwrap_or_default()` handles clock error gracefully (returns Duration::ZERO).
- Fallback to `event.timestamp_ns` when clock returns 0 prevents rejecting all events.
- Same pattern applied in `crates/gossip/src/sync.rs` line 324.

**Verdict: CORRECTLY IMPLEMENTED.** Both the truncation and clock-failure edge cases are handled.

- Padding uses `Hash32::ZERO` instead of last-leaf duplication (prevents second-preimage via leaf duplication).

**Verdict: CORRECTLY IMPLEMENTED.** Full RFC 6962 compliance. Both bridge proof and state Merkle tree use domain-separated hashing.

## SECTION 3: NEW FINDINGS

### N-01 | MEDIUM | Transaction Decode Uses Default Bincode (No Size Limit)

**File:** `crates/types/src/transaction.rs` line 205
**Line:** `bincode::deserialize(bytes).map_err(...)`

**Description:**
`Transaction::decode()` checks `bytes.len() > MAX_DECODE_SIZE` (128 KB) at line 200-203 as a pre-check, but then calls `bincode::deserialize(bytes)` with default bincode options at line 205. Default bincode does NOT enforce internal size limits during deserialization. A crafted 127 KB payload could contain a Vec length header claiming millions of elements, causing bincode to attempt a multi-GB allocation before the outer size check catches it.

This is inconsistent with `Event::decode()` and `GossipMessage::decode()` which both use `bincode::options().with_limit()`. The `StateCheckpoint::decode()` also uses limited options.

**Impact:** Potential OOM via crafted transaction payload that passes the outer length check but triggers excessive allocation during bincode deserialization. The outer check mitigates this partially (128 KB max input), but a nested Vec<Vec<u8>> within TransactionKind variants (e.g., `Deploy { code: Vec<u8>, init_data: Vec<u8> }`) could still amplify within that budget.

**Fix:**
```rust
pub fn decode(bytes: &[u8]) -> Result<Self, TransactionError> {
    use bincode::Options;
    if bytes.len() > Self::MAX_DECODE_SIZE {
        return Err(TransactionError::DecodeFailed(...));
    }
    let opts = bincode::options()
        .with_limit(Self::MAX_DECODE_SIZE as u64)
        .with_fixint_encoding();
    opts.deserialize(bytes).map_err(|e| TransactionError::DecodeFailed(e.to_string()))
}
```

**Severity:** MEDIUM — mitigated by the outer 128 KB length check, but inconsistent with hardened decode paths elsewhere.

### N-02 | MEDIUM | DAG Pruning TOCTOU Between Read and Write Phases

**File:** `crates/hashgraph/src/dag.rs` lines 744-831

**Description:**
`prune_before_round()` operates in three phases:
1. Phase 1: Read lock on `events` to identify candidates and retained events (lines 750-776).
2. Phase 2: Drop read lock (line 776).
3. Phase 3: Acquire write locks to remove candidates (lines 786-823).

Between Phase 1 (read) and Phase 3 (write), a concurrent `insert()` call could add a new event whose self_parent or other_parent references an event in the `candidates` set. This event would pass insert validation (parent exists in the events map), but then pruning removes the parent, leaving a dangling reference.

The window is narrow (pruning is called infrequently, only after consensus advances), and the practical impact is limited because:
- Pruning only removes events from very old rounds (1000+ rounds behind).
- New events referencing such old parents would be unusual in a healthy network.
- The BFS in `can_see`/`strongly_sees` gracefully handles missing parents (returns false).

**Impact:** A concurrent insert during pruning could theoretically leave an event with a dangling parent reference, causing `can_see()` to return incorrect results for that event. In practice, this is very unlikely due to the round distance.

**Fix:** Either hold the events write lock for the entire operation (expensive but correct), or add a post-insert check that parents still exist after acquiring the write lock in Phase 3 and before committing the removal.

**Severity:** MEDIUM — theoretically possible but practically unlikely. Consensus correctness is not affected because `can_see` returns false for missing parents, which is safe (conservative).

### N-03 | LOW | FalconPublicKey/FalconSignature Use Derived PartialEq (Non-Constant-Time)

**File:** `crates/crypto/src/quantum.rs` lines 40-45

**Description:**
```rust
#[derive(Clone, PartialEq, Eq, Serialize, Deserialize)]
pub struct FalconPublicKey(pub Vec<u8>);

#[derive(Clone, PartialEq, Eq, Serialize, Deserialize)]
pub struct FalconSignature(pub Vec<u8>);
```

`FalconPublicKey` and `FalconSignature` use derived `PartialEq` which performs short-circuit byte comparison. This is a timing side-channel — an attacker who can measure comparison time can learn the prefix of a public key or signature.

In contrast, `Ed25519PublicKey`, `Ed25519Signature`, and `Hash32` all have manual `PartialEq` implementations using `subtle::ConstantTimeEq`. The Falcon types are inconsistent with this hardening.

**Impact:** LOW — Falcon keys are used for long-term validator identity, not for session tokens or MACs. The timing side-channel is primarily relevant when comparing secrets against attacker-controlled inputs, which is not the typical Falcon key usage pattern. However, for defense-in-depth consistency, constant-time comparison should be applied.

**Fix:**
```rust
impl PartialEq for FalconPublicKey {
    fn eq(&self, other: &Self) -> bool {
        use subtle::ConstantTimeEq;
        self.0.ct_eq(&other.0).into()
    }
}
// Same for FalconSignature
```

**Severity:** LOW — timing side-channel exists but is not practically exploitable in the current usage context.

## SECTION 4: POSITIVE FINDINGS (What Is Done Well)

1. **Consistent `#![forbid(unsafe_code)]`** across all security-critical crates (crypto, wallet, mempool, executor, gossip, sync, bridge). Zero unsafe blocks in the entire codebase.

2. **Domain separation everywhere**: Event hashes use `"cathode-event-v1:"` prefix, Merkle leaves use `0x00`, internal nodes use `0x01`, relay proofs use `"cathode-relay-v1:"`, coin flips use `"cathode-coin-v2-multi-witness"`. This prevents cross-domain hash collisions.

3. **Comprehensive constant-time comparisons** for Hash32, Ed25519PublicKey, Ed25519Signature using `subtle::ConstantTimeEq`.

4. **Defense-in-depth chain_id enforcement**: Checked at gossip layer (sync.rs), mempool (lib.rs), AND executor (pipeline.rs). Three independent barriers against cross-chain replay.

5. **Bounded data structures everywhere**: ReceiptStore (100K), checkpoint history (100), mempool known set (100K), DAG rate limits (per-creator + global), bridge liquidity cap. No unbounded growth vectors remain.

6. **TOCTOU elimination**: DAG insert performs all checks under a single write lock (lines 296-400). Mempool submit uses double-check locking pattern. Transfer locks use deterministic ordering (smaller address first).

7. **Proper key zeroization**: Ed25519KeyPair zeros signing key on drop via `zeroize::Zeroizing`. FalconKeyPair extracts secret bytes into Zeroizing wrapper on drop.

8. **Robust consensus algorithm**: BFS over both parents for `earliest_seeing_time` (CS-01 fix), lower-median for consensus timestamps (CS-02 fix), multi-witness coin flip (E-04 fix), slashed creator exclusion from voting (C-02 fix), minimum stake filter for famous witnesses.
